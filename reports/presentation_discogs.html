Pourquoi ce sujet ?

Discogs — encyclopédie mondiale du vinyl physique et numérique. Chaque ligne = un pressage individuel (vinyl, CD, fichier numérique…) avec ses métadonnées complètes. Pas une œuvre abstraite — un objet physique précis, avec son label, son pays, ses 300 exemplaires.
La musique électronique comme terrain data
  • La musique électronique n'est pas un genre — c'est un écosystème. Des labels qui pressent à 300 exemplaires, des vinyls à 200 € vingt ans après leur sortie.
  • Discogs impose de vraies contraintes data : hétérogénéité des tags, valeurs manquantes, multiples labels par release, granularité très fine.
  • Cas réel et imparfait — à l'opposé d'un dataset "propre de cours".
  • Ce projet naît d'une double passion : la musique électronique d'un côté, la data de l'autre.
3 questions centrales
1. Structure — Comment le catalogue électronique est-il organisé dans le temps, par format, pays, label et style ?
2. Séparation — Qu'est-ce qui différencie structurellement un vinyl d'une release digitale ? Les métadonnées suffisent-elles ?
3. Collectabilité — Quels signaux font qu'un vinyl devient rare et désirable ? Peut-on les mesurer et les prédire ?

Enrichissement — API Discogs

Partie 5 · collecte des signaux marchands sur 2 139 vinyls ciblés
Pourquoi l'API ?
  • Le dump CSV ne contient pas les signaux marchands en temps réel : want, have, prix de revente.
  • L'API Discogs permet d'interroger chaque release individuellement pour récupérer ces données.
  • Ces signaux sont le cœur de la modélisation de collectabilité.
Contraintes techniques traitées
  • Rate limiting : 1 requête / seconde (respect des CGU Discogs)
  • Cache .pkl résumable : interruption possible, reprise automatique
  • ThreadPoolExecutor : collecte séquentielle avec timeout
  • Sauvegarde d'urgence : signal + atexit → aucune donnée perdue

Synthèse — 3 réponses, 3 questions

Ce que les données nous ont appris
1 — Structure du catalogue · Les patterns culturels sont robustes et mesurables : UK/Drum n Bass, Allemagne/Techno, comeback vinyl 2010s sont visibles dans 4,85M de lignes. Les données confirment l'histoire.
2 — Vinyl vs Digital · La séparation est structurelle, stable sur 25 ans, portée par le label et le pays. Sans connaître le format, un modèle le prédit à 83% de précision.
3 — Collectabilité · 67% de la variance expliquée · ROC-AUC 0.921. La rareté × la réputation du label × l'ancienneté ne s'additionnent pas — elles se multiplient. Baby Pop (178 €, ratio 29) vs Taylor & Smith (6 €, ratio 0) — le modèle le voit.
Limites à garder en tête
  • Enrichissement partiel : 2 139 releases sur 4,85M → conclusions solides sur ce sous-ensemble, généralisation prudente.
  • Snapshot : want/have évolue en permanence — une release peut devenir culte du jour au lendemain.
  • style_first seulement : la nuance multi-style est perdue (une release peut appartenir à 5 styles).
  • Biais de sélection : les labels ciblés en Partie 5 ne représentent pas tout le catalogue.
Ces limites bornent les résultats mais ne les invalident pas. Elles ouvrent des axes d'amélioration concrets : plus d'enrichissement API · validation temporelle Partie 5 · TF-IDF multi-label.

Backup — Choix des modèles : le raisonnement

Partie 4 — Pourquoi LogReg > RandomForest ?
  • Données très creuses après OHE (milliers de labels/pays encodés). La régression logistique gère bien les espaces sparse.
  • Le RandomForest divise les features une par une à chaque nœud → moins efficace sur des features très creuses.
  • LogReg + régularisation L2 → meilleure généralisation sur les classes rares.
  • Pour la classification binaire avec des features catégorielles dominantes, LogReg est souvent optimal.
LogReg = interprétable + régularisé + efficace sur sparse. Les coefficients donnent directement le sens de chaque variable.
Partie 5 — Pourquoi GBR > RF en classification ?
  • Dataset petit (2 139 releases) → le boosting séquentiel s'adapte mieux aux petits samples.
  • Le Gradient Boosting corrige itérativement les erreurs du modèle précédent → précision croissante.
  • Le Random Forest fait la moyenne de 200 arbres indépendants → plus robuste sur gros volumes, mais moins précis sur petits.
  • Pour la régression, le RandomForest reste meilleur (R²=0.674 vs 0.643) : la variance des prédictions est mieux capturée en bagging sur ce problème.
Il n'y a pas de "meilleur modèle universel" — le contexte (taille du dataset, type de features, cible) détermine le choix.

Gestion du rate limiting
  • API Discogs : 60 req/min pour utilisateurs authentifiés.
  • time.sleep(1) entre chaque appel → respect strict des CGU.
  • Temps total de collecte : ~36 minutes pour 2 139 releases.
  • Clés stockées dans .env.local (exclu du repo GitHub).
Pourquoi 2 139 releases ?
  • Sélection des releases vinyl des labels Electronic ciblés (URLs Discogs en 5.1).
  • Représentatif du marché collector Electronic : labels indépendants, pressages limités.
  • Suffisant pour entraîner les modèles de Partie 5 (CV 5-fold stable).
